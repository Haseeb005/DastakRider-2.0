# Dastak Rider App — Full Feature & Code Documentation

## Overview

The Dastak Rider App is a web-based delivery-partner interface available at `/rider`. It is built with React (TypeScript) on the frontend and Express.js on the backend. Riders log in with phone + password, go online/offline, pick up available orders, update delivery statuses, track earnings, and view delivery history.

**URL:** `https://dastakbites.com/rider`  
**Backend file:** `server/rider-routes.ts`  
**Frontend file:** `client/src/pages/rider-app.tsx`  
**Database:** MongoDB (`users` collection for riders, `orders` collection)

## 1. Authentication

### Features
- Login with phone number + password
- Register as a new rider (name, phone, password, city, vehicle type)
- Session-based authentication (server-side session cookie)
- Logout clears the session

### Backend Endpoints

#### POST `/api/rider/login`
```ts
Body: { phone: string; password: string }
Response: RiderProfile object

// Checks users collection (type:"rider") first, then legacy riders collection
// Stores riderId in session: "u_<mongoId>" for users, plain uuid for legacy
```

#### POST `/api/rider/register`
```ts
Body: {
  name: string;
  phone: string;
  password: string;
  city: "Lahore" | "Karachi" | "Islamabad" | "Rawalpindi" |
        "Faisalabad" | "Multan" | "Peshawar" | "Quetta" | "Sargodha";
  vehicleType: "bike" | "car";
}
Response: 201 + RiderProfile object

// Stores in users collection with type:"rider"
// Checks for duplicate phone numbers
```

#### POST `/api/rider/logout`
```ts
// Clears riderId from session
Response: { ok: true }
```

### Frontend Code

## 3. Available Orders Tab

### Features
- Lists all orders with status `"Admin Accepted"` that have no rider assigned yet

- Tapping an order shows all details: customer name, address, phone, items, payment type, order total, rider fare
- One-tap **Accept Order** button
- Real-time audio alert when a new order appears (ascending arpeggio melody, plays for 10 seconds)
- Browser push notification on new orders
- Auto-refreshes every 10 seconds

```ts
    { freq: 1319, dur: 0.20 },   // E6
  ];
  // Schedules oscillators with gain envelopes; alternates "triangle"/"sine" timbre
  return () => { ctx.close(); }; // call to stop early
}
```

## 4. Active Deliveries Tab

### Features
- Shows all orders currently assigned to this rider that are not yet delivered
- Two action buttons depending on current status:
  - **"Admin Accepted" / "Rider Accepted"** → "Picked Up from Restaurant" button
  - **"Rider Picked Up"** → "Mark as Delivered" button
- Tapping an item shows full order details
- Auto-refreshes every 10 seconds
- Badge count on the bottom nav tab

### Backend Endpoint

#### GET `/api/rider/orders/active`
```ts
// Requires: session riderId
// Returns orders where riderId=this rider AND status IN ["Admin Accepted","Rider Accepted","Rider Picked Up"]
Response: Array<RiderOrder>  // same shape as available orders
```

#### PUT `/api/rider/orders/:id/status`
```ts
Body: { status: "rider_picked_up" | "delivered" }
// Status mapping:
//   rider_picked_up → "Rider Picked Up"  (pickup confirmed)
//   delivered       → "Delivered"         (order complete)
// On "Delivered":
//   - Sets paidToRider: false
//   - Sets timeWhenDelivered (PKT time string)
//   - Increments rider's orderCount in users collection
//   - Fires WebSocket live-update event so customer's order page refreshes
Response: { ok: true }
```

### Status Flow
```
Admin Accepted
     ↓  (rider accepts)
Rider Accepted
     ↓  (picked up from restaurant)
Rider Picked Up
     ↓  (delivered to customer)
Delivered
```

## 5. Order Detail View (Item Detail Sheet)

### Features
- Tapping any order item opens a modal with full detail
- Shows: base item name, quantity, size, price per unit, total price
- **Deal detection**: items with selections formatted as `"Burger Deal (Coleslaw, Small Drink, Fries)"` are parsed and each selection shown as a numbered list
- Separate display for online-paid vs COD orders

### Parse Logic
```ts
function parseItemName(fullName: string): { baseName: string; selections: string[] } {
  const match = fullName.match(/^(.*?)\s*\((.+)\)\s*$/);
  if (match) {
    return { baseName: match[1].trim(), selections: match[2].split(",").map(s => s.trim()) };
  }
  return { baseName: fullName, selections: [] };
}
// Example:
// "Zinger Burger (Extra Sauce, No Pickles)" →
//   baseName: "Zinger Burger"
//   selections: ["Extra Sauce", "No Pickles"]
```

## 7. GPS Location Tracking

### Features
- Rider app pushes GPS coordinates to the server while on a delivery
- Customer's order tracking page receives real-time rider location
- Location data expires after 60 seconds (stale detection)
- In-memory store — no database required for location data

### Backend Endpoints

#### POST `/api/rider/location`
```ts
// Requires: session riderId
Body: { orderId: string; lat: number; lng: number }
// Stores in in-memory Map<orderId, { lat, lng, riderId, ts }>
// Fires WebSocket event so customer map refreshes instantly
Response: { ok: true }
```

#### GET `/api/orders/:id/rider-location`
```ts
// Public endpoint — called by the customer's order tracking page
// Returns 404 if no location or if last update > 60 seconds ago
Response: { lat: number; lng: number; ts: number }
```

### In-Memory Store
```ts
const riderLocations = new Map<string, {
  lat: number;
  lng: number;
  riderId: string;
  ts: number;  // Date.now() at time of update
}>();
```

## 8. Real-Time Updates (WebSocket)

When a rider updates an order status or pushes GPS location, the server fires:
```ts
storage?.notifyChange?.({
  collection: "orders" | "riderLocation",
  operation: "update",
  id: orderId
});
```
This broadcasts to all connected WebSocket clients. The customer's browser receives this event and invalidates the relevant React Query cache keys, causing the order status and map to refresh without a page reload.

## 12. Admin Setup Requirements

For a rider to receive orders, the admin must:

1. **Set `riderZones`** — array of zone names matching the `zone` field on orders (e.g. `["DHA Phase 5", "Gulberg III"]`). Only orders in these zones appear in the rider's available list.
2. **Set `tillNoonFare`** — the per-delivery fare (Rs.) the rider earns. This is copied onto the order when the rider accepts it.
3. **Set order status to `"Admin Accepted"`** — orders only appear in the available list once admin has accepted them.

## 13. Security Notes

- Rider sessions use the same Express session middleware as the customer app (cookie-based)
- `riderId` stored in session as `"u_<mongoObjectId>"` for new riders
- All rider endpoints call `requireRiderId()` which returns 401 if session is missing
- Passwords stored as plain text in the `users` collection (should be upgraded to bcrypt)
- Legacy `riders` collection uses bcrypt hashed passwords
